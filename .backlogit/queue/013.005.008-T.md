<!-- BEGIN:description -->
F.5 prerequisite (added 2026-07-04 after the 013.005.003-T investigate-first halt). The intercom tool-serving surface — IntercomServer + src/mcp/tools/* + sse.rs — is currently the HTTP MCP endpoint that spawned agents, INCLUDING ACP-mode agents (main.rs:365-368; HITL-003/FR-032), call to invoke intercom tools: accept_diff, ask_approval, check_auto_approve, forward_prompt (transmit), heartbeat, recover_state, remote_log, set_operational_mode, wait_for_instruction (standby). This tool-serving lives under the MCP-specific src/mcp/ path and is coupled to rmcp. Per operator direction and proper API design, MCP and ACP should each own their endpoint; ACP must have its OWN native tool path rather than reaching through MCP-path code (the fact it was never factored into a shared resource path is evidence it is MCP-specific). Refactor/rebuild the tool-serving functionality into an ACP-native tool surface so ACP agents invoke intercom tools WITHOUT the HTTP MCP endpoint. Per ADR-0016: tools that map to standard ACP messages route over the ACP stream (session/request_permission, session/update, session/cancel); intercom-specific tools with no ACP equivalent ride standard extension points (session/update variants, _meta) or a clearly-labeled local ACP extension — never re-encoded as bespoke wire methods.
<!-- END:description -->

<!-- BEGIN:implementation-notes -->
Recommend an impl-plan before building — this is design-heavy and likely decomposes into subtasks. Start by cataloguing every tool in src/mcp/tools/ and classifying each: (a) already handled as a standard ACP message in src/acp/reader.rs, (b) needs a new ACP-native handler/extension, or (c) genuinely dead (MCP-only, safe to drop). Then implement ACP-native handlers for the tools ACP agents actually use. Consumes ADR-0016 (wire conformance). This task is the prerequisite that must land before F.5-T4 (013.005.003-T) can safely delete src/mcp/*.
