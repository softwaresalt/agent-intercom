## Key Decisions

1. **`copilot.exe --acp` (not `--stdio`)**: The Copilot CLI uses `--acp` flag for ACP mode. `--stdio` produces zero output.
2. **JSON-RPC 2.0 protocol**: ACP is JSON-RPC 2.0 over NDJSON. Full handshake: initialize â†’ result â†’ initialized â†’ session/new â†’ session/prompt. `protocolVersion` is a NUMBER (1), not string.
3. **`session/prompt` format**: Prompt is an array `[{type: "text", text: "..."}]`, not a string. Requires `sessionId` from `session/new` response.
4. **Value-based session/update parsing**: Replaced typed structs with `serde_json::Value` extraction because update types have incompatible JSON shapes (single object vs array content).
5. **agent_session_id in-memory + DB**: Stored both on AcpDriver (fast lookup) and in DB (persistence across restarts).
6. **Debounced text aggregation**: 2-second debounce per session to aggregate word-by-word `agent_message_chunk` fragments before posting to Slack. Flush interval checks every 500ms.
7. **Termination posts to session thread**: Changed SessionTerminated handler to look up session's channel_id/thread_ts instead of using global config channel.

## Failed Approaches

- **Ready-signal wait**: Initially expected the agent to emit a "ready" signal on stdout. Copilot CLI produces zero stdout until it receives an initialize message. Removed ready-signal wait entirely.
- **`--stdio` flag**: Initially configured `host_cli_args = ["--stdio"]`. Copilot CLI's `--stdio` mode is for MCP, not ACP. Changed to `["--acp", "--yolo"]`.
- **Strongly-typed session/update parsing**: Used `SessionUpdatePayload`/`SessionUpdateContent` structs but `tool_call_update` has a content ARRAY vs `agent_message_chunk` which has a content OBJECT. Switched to Value-based extraction.
- **`prompt/send` method**: AcpDriver originally sent `{"method": "prompt/send"}` which is a made-up method. The actual ACP protocol uses `session/prompt` JSON-RPC format.
