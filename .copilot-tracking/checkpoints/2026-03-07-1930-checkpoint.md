## Key Decisions

1. **`parse_risk_level` uses 2-arm match (not 3)**: `"high"` and `"critical"` are explicit; `"low"` falls through to wildcard `_` → `RiskLevel::Low`. This avoids `clippy::match_same_arms`. "low" and unknown values both default to Low, which is the safe default per FR-011.

2. **Agent `request_id` used as `approval.id`**: After `ApprovalRequest::new()` generates a UUID, we override `approval.id = request_id.to_owned()`. This ensures ACP JSON-RPC correlation: `clearance/response` sends `"id": request_id` which the agent can match to its original `clearance/request`. Also enables `approval_repo.get_by_id(request_id)` in the Slack approval handler.

3. **`post_message_direct()` not `enqueue()`**: Clearance messages use the direct (non-queued) Slack post to capture the returned `SlackTs`. This timestamp is stored for subsequent `chat.update` button replacement after operator decision.

4. **`#[allow(clippy::too_many_lines, too_many_arguments)]`**: Both `run_acp_event_consumer` (108 lines) and `handle_clearance_requested` (106 lines, 8 args) exceed pedantic limits. The complexity is inherent to the multi-step pipeline and cannot be reasonably reduced without losing clarity.

## Failed Approaches

- **Using `approval.id` as driver key**: Initial implementation registered with `approval.id` (UUID) and used it as the Slack button value. Adversarial review identified this breaks ACP `clearance/response` correlation — fixed by using `request_id` instead.
- **`TempDir::into_path()`**: Deprecated in tempfile 3.10. Fixed by keeping `TempDir` alive in scope and using `.path()`.
