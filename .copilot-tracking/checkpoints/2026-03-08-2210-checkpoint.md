## Key Decisions

1. **DEFERRED Fix 3 (no timeout)**: Adding a timeout requires a new `GlobalConfig` field (`fallback_reply_timeout_seconds`) and architecture changes across three handlers. Deferred as TODO(F-20) — acceptable for initial delivery.

2. **DEFERRED Fix 4 (no termination cleanup)**: Cleanup requires session lifecycle hooks and a map restructure to store `session_id` alongside the sender. Deferred as TODO(F-20).

3. **`pub` vs `pub(crate)` for thread_reply functions**: Kept as `pub` because external test crates (`tests/unit/`, `tests/integration/`) cannot access `pub(crate)` items. The architectural recommendation to use `pub(crate)` is documented but not applied.

4. **authorized_user_id stored in map, not passed as parameter**: The fix stores the button-clicker's user ID at `register_thread_reply_fallback` time, so `route_thread_reply` no longer needs an external authorized_user_id argument. Only the user who originally clicked the button can complete the fallback.

5. **FR-022 partial compliance**: Buttons are replaced with "⏳ Awaiting thread reply…" at fallback activation. A final-decision replacement after the reply is received is NOT possible from the spawned task (no Slack message coordinates available). Documented in code comments.
